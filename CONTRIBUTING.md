# Contributing

Open Maintainer is a Bun and TypeScript monorepo. Keep changes scoped to the
surface you are working on, and use the existing app and package boundaries
under `apps/` and `packages/`.

## Setup

Install dependencies from the repository root:

```sh
bun install --frozen-lockfile
```

Useful local entrypoints:

```sh
bun run cli --help
bun run dev:api
bun run dev:web
bun run dev:worker
```

## Pull Request Workflow

1. Read the target file and at least one related caller, test, type definition,
   or user-facing doc before editing.
2. Make the smallest bounded change that addresses the issue.
3. Add or update tests when behavior, generated output, API contracts, consent
   gates, webhook handling, or context PR behavior changes.
4. Update docs when CLI flags, generated outputs, setup, dashboard behavior,
   GitHub Action behavior, Docker Compose wiring, or smoke gates change.
5. Include the validation commands you ran in the PR notes, and mention any
   skipped checks with the reason.

Avoid unrelated formatting, dependency churn, lockfile churn, generated-file
updates, or public behavior changes outside the requested scope.

## Quality Gates

Run the narrowest relevant checks while iterating. Before broad implementation
work is opened or merged, use the full gate when services are available:

```sh
bun lint
bun typecheck
bun test
bun run build
bun run smoke:mvp
docker compose up --build
bun run smoke:compose
```

Common scoped checks:

```sh
bun run --cwd apps/cli build
bun run --cwd apps/api build
bun run --cwd apps/web build
bun run --cwd apps/worker build
bun run tests/smoke/mvp-demo.ts
bun run tests/smoke/compose-smoke.ts
```

## Safety Notes

- Do not send repository content to model-backed generation unless the command
  includes explicit consent such as `--allow-write`.
- Preserve existing generated context files by default. Use `--force` only when
  overwriting generated artifacts is intentional.
- Treat GitHub App credentials, webhook verification, repository-content
  transfer to providers, context PR writes, Docker Compose service wiring, and
  lockfiles as high-risk changes.
- Do not edit `apps/web/next-env.d.ts`; it is generated by Next.js.
